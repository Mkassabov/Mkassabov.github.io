Singlepage NoScript Websites

For the better part of the last decade, the dominant method by which a web frontend was created was through a JS framework. The whole goal of these frameworks was to create an easy way to sync the UI with the state (the information held by javascript). Frameworks do a really good job of achieving that task. However, many frontend developers have started using frameworks for their other features: components, single page websites, and html templating. While frameworks do a good job of implementing these features, they are not needed for many smaller projects and often add more complexity. This complexity can often lead to web pages that are slower and more resource intensive than they should be. This can especially be a problem for users running on extremely low-end devices, or even privacy conscious users who choose to run a no-script browsing experience.

Many developers might point out that the kinds of users making the choice to use no-script should expect things not to work. I definitely agree with that statement, however that doesn’t mean that there shouldn’t be somebody out there supporting these users. At the very least, it teaches developers to rely on CSS and HTML much more than just falling back to JS whenever they can’t solve an issue. This skill becomes significantly more useful when you think about it in terms of frontend and backend instead of HTML/CSS and JS. I’ve seen a lot of web developers try to make a simple portfolio website for a client and immediately jump to JS, a Framework, and even a custom backend. This is beyond excessive!

As an example I will use a portfolio website I started building for a client, which consisted of a medium for digitally displaying their artwork, while being able to specify which ones are available for purchase, and at what price. The goal here will be to lower the cost to the lowest possible amount, the target to beat will be the $200/year that the client said they would be charged in order to have a basic e-commerce platform on one of the popular website building platforms. At the end I’ll compare my cost and what I offered compared to one of these popular services, both in terms of yearly cost and upfront cost.

No Custom Backends

There is almost no need for a custom backend hosting service for a personal portfolio / basic e-commerce website. Sometimes that added complexity even adds potential to make things worse for the client. Let's take a minute to think about where there might be a reason to add a custom backend, and then solve each problem individually:

User Added Content

There are multiple ways around this. I’ve seen a lot of people use something like Blogger to make this possible. I personally dislike Blogger, though it is definitely a good option. What I normally offer is to set everything up to run on a google drive based system. The only reason I prefer Google Drive is because a spreadsheet can often act as a simple pseudo-database for clients. With the portfolio example I mentioned the client wanted to be able to upload her new artwork whenever they wanted to be able to upload new artworks whenever they wanted without having to go through me. Since each artwork was also unique there had to be an option to show whether it was available for purchase and if so what its price was. This was very easy to manage through a Google Sheet and it provided a simple enough interface that the client was already familiar enough. In order to update the information on the site after adding to the sheet there are 2 possible methods. One is to just load the content from the spreadsheet into the website using JS, the other option is to add an option to the spreadsheet menu (via Google Scripts) that parses the information into a basic HTML template and pushes it to github (this was more no-script friendly and seemed to update faster than the Google Spreadsheet hosting). Blogger and Google Docs are only 2 of multiple possible pseudo-backend solutions that are free and pretty much just as easy to use as some of the online tools I’ve tried.

Commerce

This is where things might get a little bit more difficult to do without a real backend. For this particular example, the client wanted to be able to approve each sale personally. Both to approve the seller and to make sure they hadn’t sold the artwork in person. The solution I came up with was a combination of HTML forms, emails, and a 3rd party payment processor. HTML forms were the obvious choice on the frontend since it was a no-script friendly way of having the user return information to my client. Remember HTML forms were around before JS, backwards compatibility for the win! These forms included the artwork the user wanted to purchase, their name, email, etc. These forms were sent to netlify, where the website was hosted, and emailed back to the client. From there the client could either email back the user that they had either sold the artwork, didn’t want to sell to them, or they could send them an invoice through Stripe. No need for any advanced custom backend, and it's all free.

JS Frameworks

While there are plenty of good uses for JS Frameworks, sometimes they are used exclusively for the extra tools they are bundled with. Using Vue as an example. You need page routing to make a single-page website? Vue Router. You want to make everything into a component for organization? Vue has components. You want scoped CSS? Vue has you covered. But what if you need all that but don’t need the UI-State management offered by Vue. Vue will lead to a bloated and slower than it needs to be website. It’s not just vue either, this happens with most frameworks. Frontend developers seem to have forgotten that before frameworks there already were solutions to routing, and to scoped css, and to components.

Scoped CSS and Components

Web Components are a great solution to components. There is also the possibility to add custom html tags using JS. However, all of this seems excessive. Why do I need to add so much JS in order to have a styled button? A front end developer should be able to style the button themselves, or at the very least copy the style from the component and apply it to their button without all the unnecessary JS. Components can be extremely useful for larger, more interactive elements. But you really don’t need one for a styled button.

CSS scoping has been around for a while in the form of CSS frameworks like SASS. What happened to just adding a div with a certain class and then scoping all your CSS within that class using SASS. If you think that the templating tools provided by frameworks add significant improvements to the way you write HTML, then you are an HTML templating tool that doesn’t rely on frameworks. Between emmet abbreviations and templating languages like PUG or HAML, writing HTML is simpler than ever.

Routing

Writing a proper router for a single page application can be difficult. There are tutorials on the internet for writing a router, and once you write one you can reuse it over and over again until it no longer suits your needs. A simple router can be made in 50 lines! It can be a fun task to try and it would make a great beginner project since it shows a healthy amount of understanding about how the browser works without requiring an extreme amount of knowledge about JS.

The bigger question here is how can one go about making a router that works on a no-script page? That sounds impossible right? Well it isn’t. At the very core a router only needs a few things:

  1. A way to manipulate the URL with user input. This can be done with hashlinks
  2. A way to take the user to a different view depending on the URL. Once again, hash links to the rescue!
  3. A way to hide a specific view from the user when it's not supposed to be shown. This is where some CSS magic is required.

Normally hash links just take you to a specific element when they are clicked on. So that's the routing part all taken care of. So how can you hide the view from a certain user using CSS? Using parallel columns. If you think of a website as essentially one giant column, each view is another column displayed parallel to that one. Locking the users ability to scroll horizontally allows them to only navigate between views by using hashlinks.

The only problem with routing with hash links and css is if you want to pass information into JS or know what view the user is currently on. This can be done fairly simply by replacing all the hash links with click events that talk to your JS router. (This has the added benefit of automatically being disabled without issues for any of your no-script users).

Comparison

All of these issues that can be handled by frameworks can also be handled within HTML, CSS, and some trickery, with the occasional JS. This way of thinking can save the client money on the backend and increase their performance on the frontend. And all the setup side on the developer can be done once and then reused as a template for every client. Now that we’ve purged a bunch of unnecessary content from the clients website, where does that leave us in terms of features and cost? My upfront cost is not free, but yearly costs are significantly cheaper!
